# useReducer Audit

Audit date: 2026-06-04

Scope: React Doctor `prefer-useReducer`, `no-cascading-set-state`, `no-chain-state-updates`, `no-derived-state`, and `exhaustive-deps` findings, with extra attention on merchant, withdrawal, security, game, dashboard, deposit, and order flows.

## Standard Used

This audit uses official React guidance as the decision standard:

- [`useReducer`](https://react.dev/reference/react/useReducer): reducers keep update logic inside a pure reducer and return the next state only.
- [`Extracting State Logic into a Reducer`](https://react.dev/learn/extracting-state-logic-into-a-reducer): reducers are helpful when many event handlers modify state similarly, state logic is complex, or isolated reducer tests would improve confidence.
- [`Choosing the State Structure`](https://react.dev/learn/choosing-the-state-structure): group state that changes together, avoid contradictory state, and remove redundant or duplicated state.
- [`Scaling Up with Reducer and Context`](https://react.dev/learn/scaling-up-with-reducer-and-context): use reducer plus context only when state and dispatch need to be shared across a tree.
- [`exhaustive-deps`](https://react.dev/reference/eslint-plugin-react-hooks/lints/exhaustive-deps): missing dependencies cause stale closures; if an effect fights the linter, restructure the code rather than suppressing the warning.

## Files Where `useState` Should Remain

- `src/app/ToastProvider.tsx`: queue state is simple; cleanup ref issue is separate.
- `src/features/auth/AuthTurnstile.tsx`: already uses a reducer for widget state.
- `src/hooks/useElementSize.ts`: one measurement state plus observer ref.
- `src/pages/merchant/LiquidityPage.tsx`: small form, reducer not yet justified.
- `src/pages/ProfilePage.tsx`: simple fetch state; use one state object if needed.

## Derived or Redundant State to Remove Instead

- `src/pages/auth/ApproveLoginPage.tsx`: replace `consuming` plus `consumeError` with a single consume status/error union.
- `src/pages/auth/MagicLinkPage.tsx`: same token-consume status cleanup.
- `src/pages/auth/VerifyEmailPage.tsx`: separate token-consume status from resend form state.
- `src/pages/auth/WithdrawalMfaPage.tsx`: replace separate draft amount/address mount setters with one lazy `draftDetails` object.
- `src/pages/merchant/DepositsPage.tsx`: initialize replay window timestamps lazily instead of mount-effect setters.
- `src/pages/merchant/LiquidityPage.tsx`: if touched, consider deriving `dirty` from a baseline snapshot rather than maintaining a separate boolean.

## Implementation Notes for Future Refactors

- Do not put API calls, socket calls, storage reads/writes, timers, navigation, analytics, or toast calls inside reducers.
- Use TypeScript discriminated unions for reducer actions.
- Keep reducers small by domain; avoid one screen-wide reducer for unrelated server data, form state, and row actions.
- Prefer user/system event names for actions, such as `FIELD_CHANGED`, `SUBMIT_STARTED`, `SUBMIT_SUCCEEDED`, `MFA_FAILED`, or `RESET`.
- Preserve current loading, error, accessibility, idempotency, and validation behavior before optimizing render counts.
- Add direct reducer tests before moving UI code to the reducer, then add user-flow tests for the screen.
